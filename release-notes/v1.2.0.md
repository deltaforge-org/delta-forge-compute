## What's Changed

The DeltaForge compute node: the query-executing process. It connects to a DeltaForge control plane and runs queries on a single node. Scale out by running more nodes, or let the control plane's autoscaler pool run them for you.

### Scale-to-zero autoscaler pool

- The compute node is now the unit the control plane's per-slot autoscaler pool warms and drains. A node can sit idle at zero cost and be brought up on demand when a query needs it, then drained back down when the pool goes quiet.
- Draining is session-aware: a node that is asked to stand down finishes its in-flight work and keeps long-running result streams alive with keepalives instead of cutting them off mid-query.
- Nodes register and heartbeat by a stable registration identity, so a slot that restarts re-attaches cleanly rather than leaving a stale entry in the pool.

### The platform runs on compute nodes too

- The desktop and cloud platform route through a compute node exactly like the drivers and CLI. The same node binary serves the desktop's co-located worker, a self-hosted fleet, and a cloud autoscaler slot from one code path.
- A node advertises a client-reachable address so callers dial it directly over the deployment's public endpoint, with no per-node host configuration in the cloud.

### Multi-cloud and ambient credentials

- A compute node joins the cluster using the installed license key as its join credential, so a scale-from-zero slot carries a valid credential the instant it starts, with no token to mint at deploy time.
- On AWS and GCP the node reads object storage through the ambient credential chain (ECS task or pod role, GCP service account) with no static keys, matching the existing Azure managed-identity path.

### AI models

- Nodes execute the new AI SQL functions (`AI_GENERATE`, `AI_GENERATE_EMBEDDINGS`, `AI_EXTRACT`, `AI_CLASSIFY`, `AI_SUMMARIZE`, `AI_TRANSLATE`, `AI_FIX_GRAMMAR`, `AI_ANALYZE_SENTIMENT`), invoking the registered model per row as part of query execution.

### Graph on cloud storage

- Graph CSR builds run directly against cloud object stores (`abfss://`, `s3://`), with sidecar I/O routed through the cloud-aware filesystem.

### Operational robustness

- Live worker trace push, plus a writable trace-directory fallback for nodes running with a read-only working directory.

### Aligned to engine 1.2.0

- Versioned to the 1.2.0 platform release.
